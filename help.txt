[2.4] Form components use Math.random() for IDs (SSR hydration risk)
      Severity: MEDIUM
      Files:
        - src/components/ui/Select.tsx:22
        - src/components/ui/Textarea.tsx:18
        - src/components/ui/Checkbox.tsx:12

      `Math.random()` produces different values on server vs client,
      which causes React hydration warnings. React 18+ ships `useId()`
      for exactly this purpose.

      Fix:
        import { useId } from 'react';
        // inside component:
        const fallback = useId();
        const selectId = id || fallback;

        - src/app/dashboard/page.tsx          (18 calls)
        - src/app/admin-os/page.tsx           (14 calls)
        - src/app/seller-hub/page.tsx         (8 calls)
        - src/app/dashboard/addresses/page.tsx (4 calls)

      Fix: a `<ToastProvider />` already exists in
      src/components/ui/ToastProvider.tsx. Wire it up in layout.tsx and

      confirm() with the existing Dialog/Modal component for Yes/No
      prompts; replace prompt() with a small input modal.

[2.6] 19 leftover `as any` casts on src/app/page.tsx
      Severity: LOW (works at runtime, hides typing benefits)

      The casts exist because `dict` (return of getDictionary) is typed
      as the union `typeof en | typeof ar` but the page accesses keys
      that exist only on en. The simplest fix is to widen the helper:

        // src/lib/i18n/server.ts
        export async function getDictionary(): Promise<typeof en> {
          ...
          return isArabic ? (ar as typeof en) : en;
        }

      Then drop every `(dict as any).Foo` on page.tsx — they become
      `dict.Foo`.

[2.7] WishlistButton receives `item as any` in two places
      Severity: LOW
      Files: src/app/page.tsx:131,207

      Same root cause as 2.6 — the local `hotDeals` / `pickedForYou`
      arrays are mapped from Product but get extra display fields. Define
      a local `type DisplayProduct = Pick<Product, ...>` and use it.

[2.8] Hardcoded "04:23:18" countdown string on homepage
      Severity: LOW
      File: src/app/page.tsx:116

      The hot-deals badge says "Ends in 04:23:18" as static text. Replace
      with the new `<CountdownTimer />` component pointing at the
      flashSaleEndsAt of the soonest-ending featured product.

[2.9] Deploy workflow runs `prisma db push` not `prisma migrate deploy`
      Severity: HIGH (data loss risk in production)
      File: .github/workflows/deploy.yml:70

      `db push` blindly synchronises the live schema and CAN drop columns
      / tables when they no longer exist. In production the only safe
      command is `npx prisma migrate deploy`. Combined with the missing
      migrations folder (§1.1) this is a foot-gun.

[3.1] DictKey imported but unused

[3.2] passwordHash: '' on account deletion
      File: src/app/api/account/delete/route.ts:23
      Bcrypt.compare with an empty hash always fails (good), but storing
      an empty string means a future code path that tries to
      re-authenticate the soft-deleted user would crash. Use a
      cryptographically impossible hash sentinel:

        passwordHash: '$2b$12$DELETED_USER_DELETED_USER_DELETED_USER_DELETED_USER',

      so any compare attempt fails fast and the row is obviously
      anonymized.

[3.3] DEBUG_BYPASS in src/app/actions/seller.ts
      File: src/app/actions/seller.ts:16
      Code: `if (process.env.NODE_ENV !== 'production' && userId.startsWith('debug-'))`

      Already gated to non-prod, but still risky. Recommend deleting it
      entirely and using a proper admin user seed instead.

[3.4] session.user.email! — non-null assertion
